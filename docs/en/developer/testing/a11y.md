# Accessibility testing

Accessibility testing (abbreviated as **a11y**) verifies that websites and admin interfaces are usable by all users, including those with disabilities. The tests are performed automatically using the `axe-core` library integrated into CodeceptJS via the `A11yHelper` helper.

## Libraries and dependencies

| Library | Purpose |
| --- | --- |
| [axe-playwright](https://github.com/abhinaba-ghosh/axe-playwright) | Integrating `axe-core` into Playwright |
| [axe-core](https://github.com/dequelabs/axe-core) | Core for detecting WCAG violations |
| [axe-html-reporter](https://github.com/lpelypenko/axe-html-reporter) | Generating an HTML report on violations |

## WCAG levels

The tests check the following WCAG tags (configured in `src/test/webapp/helpers/a11yhelper.js`):

| Tag | Description |
| --- | --- |
| `wcag2a` | WCAG 2.0 Level A |
| `wcag2aa` | WCAG 2.0 Level AA |
| `wcag21a` | WCAG 2.1 Level A |
| `wcag21aa` | WCAG 2.1 level AA |
| `wcag22aa` | WCAG 2.2 level AA |

## Configuration

`A11yHelper` is configured in `src/test/webapp/codecept.conf.js`:

```js
A11yHelper: {
    require: './helpers/a11yhelper.js',
    outputDir: '../../../build/test/allure-results',
    skipFailures: false,
    reporter: "html"
}
```

Page object `a11y` is registered as:

```js
a11y: './pages/a11y.js',
```

## Running tests

### With the npm command

```sh
cd src/test/webapp
npm run a11y
```

The script `a11y` runs all tests from the folder `tests/a11y/**/*.js` with the settings:

### Shell script with Allure report

```sh
cd src/test/webapp
./a11y.sh [parameter]
```

The script accepts an optional parameter:

| Parameters | Command | Description |
| --- | --- | --- |
| *(without parameter)* | `npm run a11y` | Runs all a11y tests |
| `current` | `npm run a11y:current` | Runs only currently selected tests |
| `pause` | `npm run a11y:pause` | Runs tests with pause (Playwright pause) |

Examples:

```sh
./a11y.sh
./a11y.sh current
./a11y.sh pause
```

The script will execute:

1. Clears previous results from `build/test/` and `build/test-results/`
2. Runs the corresponding `npm run` command according to the parameter
3. Copies `allure/categories.json` to the results
4. Generates Allure HTML report to `build/test-results/`
5. Opens a report on port `3001`

## Writing tests

Each a11y test uses the `a11y.check()` method, which runs an `axe-core` analysis of the current page and compares the number of violations to 0.

```js
Feature('a11y.nazovfunkcie');

Before(({ I, login }) => {
    login('admin');
});

Scenario('popis scenára', async ({ I, a11y }) => {
    I.amOnPage('/admin/v9/vasa-stranka/');
    await a11y.check();
});
```

### What the `a11y.check()` method does

1. Runs `I.runA11yCheck()` (from `A11yHelper`)
2. In case of violations:
   - Creates an error message in the format `IMPACT: id - popis`
   - Adds an attachment with a detailed bug report
3. Compare the number of violations to 0 using `I.assertEqual` – the test fails if there are any violations

### Sample output on failure

```txt
3 accessibility violations were detected:

- SERIOUS: html-has-lang - Ensure every HTML document has a lang attribute
- CRITICAL: label - Ensure every form element has a label
- CRITICAL: select-name - Ensure select element has an accessible name
```

## Demo page

The file `src/main/webapp/components/aceintegration/admin/a11y.html` serves as a reference accessible page for a positive test (it does not contain any WCAG violations):

- Has `skip to main content` link for keyboard users
- All images have the `alt` attribute
- Headings are hierarchically correct (`h1`, `h2`)
- Navigation has `aria-label`
- Buttons are semantically correctly labeled
