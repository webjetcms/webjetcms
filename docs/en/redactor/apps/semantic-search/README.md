# Semantic Search (RAG)

Semantic search allows visitors to find relevant pages based on the **meaning of the query**, not just keyword matches. It uses the vector database [pgvector](https://github.com/pgvector/pgvector) and vectors generated via the OpenAI API.

Optionally, it can also display a **RAG answer** above the same index - a short answer generated by AI only from the found web content. The answer will be displayed above the classic search results list.

![](rag-result.png)

## What the visitor will see

- Semantic search displays a list of relevant pages sorted by similarity in meaning to the question.
- Hybrid search combines semantic ordering with full-text matching in indexed texts.
- When RAG response is enabled, the **AI response from search** block will be displayed above the results.
- If the response cannot be constructed from indexed content, no RAG response will be displayed.

## Setting up semantic search

To run a semantic search, you need:

- Enable semantic search by setting the configuration variable `ragSemanticSearchEnabled` to the value `true`.
- Set the search type to `semantic` or `hybrid`. You can do this globally via the `searchType` configuration variable, or directly in the **Search** application.
- In hybrid mode, verify that the configuration variable `ragHybridSearchEnabled` is set to the value `true`.
- Verify that the configuration variable `luceneAsDefaultSearch` is set to the value `false`. If it is set to `true`, Lucene will be used instead of semantic search because it has a higher priority.
- Set the OpenAI API key also used for AI assistants in the variable `ai_openAiAuthKey`.
- Run indexing via the administrator interface to create vectors and populate the vector database.
- Set up an automated task `sk.iway.iwcm.rag.service.RagIndexCronTask` that processes the indexing queue.

!>**Note:** We recommend that you re-index your pages after implementing these changes. The index now also stores the page folder information (`group_id`, `root_group_l1`, `root_group_l2`, `root_group_l3`), which is used when filtering results by folders selected in the **Search** application.

## Search app settings

Semantic search is used through the **Search** application embedded in the page. The application has settings for semantic, hybrid, and RAG search in addition to the regular fields.

### Basic tab

- **Search Type** - determines what mechanism a particular embedded application will use:
  - **By configuration** - will use the global value `searchType`.
  - **Database** - will use standard database search.
  - **Fulltext/Lucene** - will use Lucene.
  - **Semantic** - will use vector search without hybrid full-text branching.
  - **Hybrid** - will use vector search along with fulltext over indexed parts of the text.
- **Add RAG Response** - enables or disables AI response for a specific application. The value **By Configuration** respects the global setting `ragAnswerAllowed`.
- **Directory** - limits the search to selected folders. For semantic and hybrid search, the restriction is applied directly above the semantic index.

If any option is marked as **currently disabled**, first enable the corresponding global configuration variable.

### Semantic Settings tab

Allows you to override global values ​​for a specific embedded application:

- **Minimum similarity** - local value for `ragSemanticSearchMinSimilarity`.
- **Minimum number of results** - local value for `ragSemanticSearchMinResults`.

An empty value means that the global configuration will be used.

### Hybrid Settings tab

Allows you to set when a full-text branch should be added to a vector search:

- **Hybrid Search Mode** - `off`, `always`, `short_query_only`, `fallback_on_low_vector` or **As configured**.
- **Maximum number of characters/words for a short query** - limits for `short_query_only` mode.
- **Threshold similarity for fallback** - threshold for `fallback_on_low_vector` mode.
- **Vector and full-text branch weights** - determine the resulting order when combining via RRF.
- **Block loading ratio** - how many text parts are loaded before being aggregated into documents.
- **Use `ILIKE` fallback for fulltext** - will use a simple text search if PostgreSQL fulltext finds nothing.

### RAG Settings Tab

The RAG response is generated only after relevant parts of the content have been found. You can set:

- **AI assistant for RAG response** - a default assistant is created automatically. Optionally, you can select your own assistant dedicated to RAG responses.
- **Minimum similarity** - soft threshold for parts of text used in generating the answer.
- **Use top K blocks** - how many of the best parts of the text enter post-processing.
- **Maximum gap between blocks** - determines whether adjacent parts of a single page will merge into a common context.
- **Maximum number of blocks** and **maximum number of characters** - limit the size of the context of the submitted AI model.
- **Maximum block character count** - prevents the creation of a merged context block that is too large.

RAG response uses only content retrieved from the semantic index. If the answer is not found in the context, the system displays fallback text instead of inventing an answer.

## Semantic index

To use semantic search, you need to have your content indexed using semantic indexing, which is available in the admin interface. For more information, see [Semantic Index](./embedding-chunks.md).
